Метрики сложности

1) метрики размера программ

- количество строк исходного кода
SLOC

- метрика М.Холстеда
n1 - число уникальных операторов программы
n2 - число операндов программы
N1 - общее число операторов программы
N2 - общее число операндов программы
n'1 - теоретическое число уник.операторов
n'2 - теоретическое число уник.операндов

Словарь программы  n= n1+n2
длина N= N1+N2
объем программы (в битах)
V = N*log2(n)
Теоретический словарь программы n'= n'1+n'2 - словарный запас, нобходимый для ее написания

Оценка V
V' = n'*Log2(n')
- описывает  потенциальный объем программы, соответствующий максимально компактному тексту программы, которая реализует данный алгоритм.

2) метрика сложности потока управления программ

- основана на анализе управляющего графа программы
- вершины графа - участки кода с последовательными вычислениями (без ветвлений и циклов)

G = (V,E)
V - вершины, соотв. операторам
E - дуги, соотв. переходам

- метрика Т.Мак-Кейба
цикломатическая сложность графа программы или цикломатическое число Мак-Кейба - характеризует трудоемкость тестирования программы

V(G) = e - n +2p
e - число дуг
n - число вершин
p - число компонентов связности (число дуг сильно связанного графа, любые 2 вершины взаимно достижимы)

- не подходит для условных и циклических конструкций

- метрика Маерса
V(G) = V(G) + h
h = 0 для простых предикатов и h=n-1 для n-местных

- метрика К.Джилба - логическая сложность - определяется как насыщенность программы  выражениями типа if-then-else (ветвлениями)

CL - абсолютная сложность - количество услоных операторов
cl - относительная сложность

cl = CL/n
n - общее число операторов

- метрика граничных значений
граф, где число входящих дуг - отрицательная степень вершины
число исходящих дуг - положительная степень вершины
принимающие - полож.степень <=1
вершины отбора - полож.степень >=2

3) метрики сложности потока данных - оценка использования, конфигурации и размещения данных в программах

- метрика обращения к глобальным переменным - связывает сложность с обращением к глобальным переменным
пары (p,r) р- модуль, r- глоб.перем-я

- метрика спена. Спен - число утверждений, содержащий данный идентификатор. Идентификатор появивший n раз - спен = n-1
При большем значении спена - усложняется тестирование и отладка.

- метрика Чепина - анализ характера использования переменных из списка ввода-вывода
С - группа управляющих переменных имеет наибольший весовой коэф.

3) метрики стилистики и понятности
- полнота документирования и комментирования исходного кода
- код разбивается на n равных кусков и для каждого из них определяется коэффициент

Уровень комментирования, насыщения
F= Nком/Nстр
F >= 0.1 - оптимальное, 1 комментарий на 10 строк кода 

Также учитывается оценка не только затрат на разработку, но и на восприятие готовой программы.

Измерительные методы применяются:
- измерение потребления программами ресурсов ПК для устранения дефектов производительности
- пред и пост- измерения параметров системы для моделирования

Условия применения измер.методов:
- наличие готовой программы
- наличие реальной выч.системы для прогона программы
- наличие АС и ПС для проведения измерений
- создание условий допустимых искажений

Два основных способа регистрации измеряемых параметров - трассирующий и выборочный.

Трассирующий способ - измеряемые параметры фиксируются в момент наступления какого-либо события
{ri(ej)}
i- ресурс, ri - значение i ресурса
ej - событие j
В качестве событий рассматриваются обычно: 
- обращения к некоторым операторам программы, программным модулям, наборам данных или устройствам системы
- прерывания процессора, наступающие при обмене данными с внешними устройствами.

Выборочный способ - измерения производятся в моменты времени tj
 tj = tj-1 + dt,   {ri(tj)}
достоинство - простота (не надо регистрировать наступление событий)

- наиболее полная характеристика тестируемой программы

Частотные профили определяют количественное распределение потребления ресурсных параметров (число раз)
Временные профили - задают распределение времен потребления ресурсов в абсолютном или относительном масштабе (мс., %)
Достоинства профилей - выявление узких мест выполнения программ в монопольном режиме.
Недостаток - отсутствие хронологии потребления ресурсов программой.
Общий недостаток профилей и трассировочных записей - зависимость от рабочей нагрузки.

Гарантии что в программе нет ошибок - не существует.

Основной причиной появления ошибок в программах является разработчик.

С ростом сложности разрабатываемого ПО:
- появление ошибок увеличивается
- актуальность задачи по их выявлению усиливается

Типы ошибок по стандарту IEEE-729-83:
- ошибка (error) - выдаются неправильные результаты
- дефект (fault) - следствие ошибок, приводит к сбоям в работе
- отказ (failure) - отклонение от функционирования или невозможность выполнять требуемые функции (неработоспособность)

Поскольку верификация ПО - это также проверка на соответствие качественным характеристикам ПО, то можно выделить:
- корректность - соответствие системы своему предназначению
- безопасность системы
- устойчивость - в случае неверных входных данных
- эффективность использования ресурсов времени и памяти
- адаптируемость - к измененям окружения
- переносимость (разное окружение, ОС)
- совместимость (версии программ)

Статический анализ ПО - анализ ПО, производимый без реального выполнения исследуемых программ.

Динамический анализ ПО - осуществляется при помощи реального выполнения программы.

На вход программы поступают последовательности данных, которые могут вызвать неопределенное поведение, тем самым позволяя обнаружить уязвимости и ошибки.

Виды динамического анализа:
- тестирование
- мониторинг
- имитационное тестирование
- профилирование

Тестирование ПО - поиск ситуаций в программном коде, в которых поведение программы становится неопределенным, неправильным и не соответствующим спецификации.

Цели тестирования:
- повышение вероятности, что приложение будет
 * работать правильно при любых обстоятельствах
 * соответствовать всем описанным требованиям
- предоставление актуальной информации о состоянии продукта на данный момент.

Стратегия тестирования "черный ящик"
- тестировщик имеет доступ только через те интерфейсы, что и пользователь, либо внешние интерфейсы.
- используются спецификации, описывающих требования к системе
Критерий покрытия = покрытия входных данных + покрытие требований.

Тестовое покрытие — это метрика, используемая для измерения качества тестирования программного обеспечения. Она показывает, какой процент кода вашего приложения был выполнен в процессе тестирования. Чем выше показатель тестового покрытия, тем больше уверенности можно иметь в том, что ваш код работает корректно и без ошибок.

Тест-дизайн - этап процесса тестирования ПО, на котором проектируются и создаются тестовые случаи (тест-кейсы) в соответствии с определенными ранее критериями качества и целями тестирования.

Виды тестирования стратегии "черного ящика":

1) эквивалентное разделение
если дан интервал допустимых значений [1,10]
берем 5 (из интервала, верное значение)
и 0 или 11 (вне интервала, неверное значение)

2) анализ граничных значений
для [1,10] - берем границы 1 и 10
и значения боль и меньше границы - 0, 11
- применим к полям, записым, файлам и пр. сущностям, имеющим ограничения

3) анализ причинно-следственных связей
ввод условий (причин) для получения ответа (следствия)
Добавляем в Корзину, заполняем поля формы, жмем Создать заказ (причина) -> новая запись в таблице БД (следствие)

4) предположение об ошибке
Тест-аналитик намеренно вводит неправильные данные или наоборот не вводит данные, чтобы вызвать ситуацию ошибки, зная требования спецификации.

Техника тестирования по стратегии "белого ящика" - позволяет проверить внутреннюю структуру программы
- анализ логики работы программы предполагает знание разработчиком внутреннего устройства ПО

Варианты тест-дизайна:
1) покрытие операторов - процентное отношение числа операторов проверяемых тестаки к общему колву
2) покрытие решений - покрытие ветвлений
3) покрытие условий и решений - условия д.б.проверены на истину и ложь
4) комбинаторное покрытие условий - условия д.б. выполнены для набора тестов не менее раза

Нефункциональные виды тестирования:

* Нагрузочное тестирование - автоматизированное тестирование имитирующее работу определенного (среднего) количества бизнес-пользователей

* Стресс-тестирование (в предельных режимах) - проверяется насколько система работоспособна у условиях стресса, оценивается способность к регенирации, (возвращению к нормальному состоянию)
- интенсивность выполнения операций
- аварийная конфигурация

* Тестирование работоспособности, стабильности или надежности - проверка работоспособности при длительном (многочасовом) тестировании со средним уровнем нагрузки

* Тестирование на отказ и восстановление - проверяется продукт с точки зрения способности восстанавливаться после возможных сбоев, возникших в связи с ошибками ПО, отказами оборудования и сети
- акцент на восстановление, сохранность и целостность данных

Корректность ПС - соответствие проверяемого объекта некоторому эталонному объекту или совокупности эталонных характеристик и правил.

Наиболее полным эталоном корректности ПС является программная спецификация.

Спецификация - формальное описание функций и данных программы, с которыми эти функции оперируют.

Корректное ПС может выдавать неправильные данные при неправильном наборе входных данных
Надежные ПС характеризуются свойством устойчивости к недопустимым наборам входных данных

Критерии качества (2 типа):
- функциональные - определяются предметной областью и функциями выполняемой программы
- конструктивные - определяются общими для всех программ свойствами

Корректность текстов программ - степень соответствия исходных текстов программ формализованным правилам языков спецификации и программирования

Конструктивная корректность модулей - соответствие структуры модулей правилам

Функциональная корректность модулей - корректность обработки исходных данных и получения результатов

Конструктивная корректность данных - определяется правилами их структурирования и упорядочения

Функциональная корректность данных - конкретизация их содержания в процессе исполнения программ

Синтаксический контроль корректности текстов программ - проверка на соответствие синтаксису языка программирования

Семантический контроль текстов программ - проверка корректности применения конструкций языка в тексте программ

Верификация (статическая проверка) - подтверждение на основе представленных объективных свидетельств того, что заданные требования полностью выполнены.
- процесс анализа результатов действий на предмет соответствия заданным для них требований.

Полноту проверки корректности ПС обеспечивают методы верификации и валидации.
Валидация устанавливает соответствие между тем, что делает программа, и тем, что нужно заказчику.

Противодействия дизассемблированию:

- архивация

- шифрование - аппаратное или программное, дизассемблирование требует предварительной дешифрования

- использование самогенерирующих кодов - генерация исполняемых кодов самой программой в процессе ее выполения

- обман дизассемблера - нарушение общепринятых соглашений, нестандартные приемы использования отдельных команд

Метод обфускации (запутывания) - для защиты программ от исследования с помощью дизассемблера - усложняется структура самой программы с целью запутывания злоумышленника.

Исследование кода вредоностных программ (180)
---------------------------------------------
Анализ вредоностных программ (ВП) - деятельность по разбору вредоностного ПО с целью определения принципов его работы, поиска методов его определения и способов противодействия воздействию на функционирование ИС.

Компьютерный вирус - программа, способная создавать свои копии (необязательно совпадающие с оригиналом) и внедрять их в файлы, системные области компьютера, сетей, а также осуществлять другие деструктивные действия, при этом копии сохраняют возможность дальнейшего распространения.
